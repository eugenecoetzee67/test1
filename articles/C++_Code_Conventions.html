Introduction

This article describes the coding standards for C++ that have currently been defined.

General Conventions

The following conventions and standards also apply to C++:

  • General Naming Conventions

Exceptions

Function, method and class names may ignore the specifications in General Naming Conventions under the following circumstances:

  • When functionality is provided that is intended to appear as if it is intrinsic to C++, then it should follow the conventions of the Standard Template Library (STL), or base C++ conventions: Examples are:
    • Another type of "cast" operator e.g. convert_cast<>().
    • Emulation of new compiler functionality not currently support by the compilers in use e.g. nullptr or std::move()
    • The "super" typedef indicating the base class of a particular class.
  • The function, method or class is intended to be used in an STL context i.e. it is an STL method or function name (such as begin()) or it is an STL class name (such as iterator).

Using exceptions

Exceptions should only be used to convey "exceptional" errors (obviously). However, what is "exceptional" depends in many cases on the context. It depends on both the reality of how the errors will occur, as well as the programming model that users of the code will expect.

For example, in a socket class it is very likely that a connection may be broken whilst reading or writing from the socket. Should the socket class throw an exception in this case? The answer is: it depends. To figure out what to do, we need to consider how the class will be used.

In most cases, the developer using the class wants to deal with the socket as if the socket is working properly - he does not necessarily want to check that every socket operation succeeds or fails. And, in practice, most socket operations will actually succeed. Also once a socket connection has been broken, the socket needs to be reinitialised before it can be used (control needs to go back to the user's socket initialisation code which, most likely, will not be part of the send or receive). In this case, it does seem to make sense to use exceptions to indicate that a connection was broken, even though it was fairly easy to anticipate.

Below are a couple of rules to help determine when and how to use exceptions.

Rules

  • Try to find a solution that does not require exceptions. The use of exceptions should only be used in those cases where it really makes sense to use them. An example where exceptions should be considered is where method calls check for an error and then return the error to the calling method. If the stack depth is quite large, then the use of exceptions may actually simplify code.
  • "catch (...)" must always rethrow the exception or terminate the application. The reason is because "catch (...)" effectively hides issues that may actually be bugs. For example, under MSVC 6, access violations are caught by "catch (...)". Therefore, if the exception is suppressed (i.e. not rethrown), then you may be leaving your application in a very dangerous state and only have the issue show up somewhere else.
  • Exceptions should only be used for cases where it is unlikely that the error will occur. I.e. exceptions should not be used for code paths that occur regularly.
  • Exceptions should not be used for expected errors. If you can anticipate the error, you should probably not be using an exception to convey the error. However, it is still possible to anticipate infrequent errors, so a judgement call is required.

Code style

if / else statements

This section supersedes the sections on "if" and "else" in C++ Coding Standard

Examples 
// if statement format
if (condition)
{
    // Some code
}

// if/else statement format
if (condition)
{
    // Some code
}
else
{
    // Other code
}

// if/else if/else statement format
if (condition1)
{
    // Some code
}
else if (condition2)
{
    // More code
}
else
{
    // Other code
}
Discussion
  • Braces are aligned under the if or else keywords.
  • Braces are *NOT* optional. They must always be used - even for "simple" one liners.
  • There is a single space between the "if" and the condition.
  • The condition has no leading or trailing spaces.
  • The condition should explicitly check for true or false, except if the condition is can clearly be read as boolean e.g. 
 if (done) // No test against true/false required
 if (CallSomeFunction() == false) // It isn't clear from the name that CallSomeFunction() actually returns a boolean.

for statements

This section supersedes the "for Statement" section in C++ Coding Standard

Examples 
// for statement format
for (declaration;  condition;  loop-expression))
{
    // Some code
}

// Simple integer for loop
for (int i = 0;  i < LOOP_COUNT;  ++i))
{
    // Some code
}

// 'STL' for loop with loop loop code on a new line
for (Container::const_iterator
    i = container.begin();  i != container.end();  ++i))
{
    // Some code
}
Discussion
  • Braces are aligned under the for keyword.
  • Braces are *NOT* optional. They must always be used - even for "simple" one liners.
  • There are *2 spaces* between the declaration & condition and the condition & loop-expression
  • When the type of loop variable is long, place the loop description on a new line indented with an extra tab. Do not put each of the loop components in different lines.


switch statements

This section supersedes the "switch Statement" section in C++ Coding Standard

Examples 
// Simple switch statement
switch (variable)
{
    case A:
        break;

    case B:  // may use braces for local scoping
    {
        break;
    }

    case C:
        break;

    default:
        break;
}
Discussion
  • Braces are aligned under the switch keyword.
  • There is a single space between the switch and the condition.
  • The condition has no leading or trailing spaces.
  • Provide a case statement for all conditions or provide a default
  • If a case or default defines local variables then braces need to be aligned under the case or default keyword.

Enumerations

This section supersedes the "Enum" section in C++ Coding Standard

Examples 
struct Example
{
    /// Actions as specified in RFC 1894
    enum RecipientAction
    {
        raUnknown,     ///< Could not be determined
        raFailed,      ///< Delivery failed
        raDelayed,     ///< Delivery has been delayed
        raDelivered,   ///< The mail was delivered
        raRelayed,     ///< The mail was relayed
        raExpanded,    ///< The mail was distributed to a distribution list
    };
};

/// The different types of ReadOnly objects that may be created
enum ReadOnlyTemplateModes
{
    /// The object becomes read-only immediately after construction.
    READONLY_AFTER_CONSTRUCTION,

    /// The object becomes read-only after the first mutable access to the object.
    READONLY_AFTER_WRITE,

    /// The object only becomes read-only after a call to ReadOnly::SetReadOnly().
    READONLY_AFTER_SET
};


Two forms of enumeration specification are allowed:

  • Camel case, with a lower-case 2 character prefix. This form should be used within classes and should not be used when a global enumeration is created.
    • Enum's do not introduce a new scope i.e. the same constant name in 2 different enum's (within the same class, namespace or global scope) results in a name clash. We use the 2 character prefix to avoid the name clash.
    • The 2 character prefix should be representative of the name of the enum.
  • All capitals, using the same conventions as constants. This convention is suitable for globally defined enum's as well as class-level enum.
    • When used as a class-level enum, the enum may be anonymous, if it makes sense to do so.
    • This is the preferred form for class-level result codes.
    • Globally defined enum's should be appropriately prefixed to avoid global name clashes.
Documentation

As shown in the examples below, enum's may be documented either on the same line (if the overall line length is short), or before each enumeration item. If the second option is chosen, a blank(ish) line should precede next documentation/comment line.

Comments

  • Use C++ style comments. This should always be done unless a source file is specifically targeting both C and C++. The reason is to make debugging easier: If only C++ comments are used, the developer can temporarily use C style comments to comment out large sections of code without other C style comments interfering.
  • Don't comment the obvious.
  • Describe the purpose of the code, not what it is doing. Comments that describe what code does tend to get out of sync with the code and then start creating confusion.
  • Prefer to write comments on their own line. This eases readability by separating the comment from the code. It also provides more real estate to provide a deeper description (as opposed to comments on the same line of code).
  • A new comment must follow a blank line. This creates a visual separation between previous code and the comment. If the previous line contained only an opening brace, then it is not necessary to precede the comment with a blank line.
Examples 
// Process input from the user until the user presses CTRL-C
while (true) // Comments on the same line as code should be avoided
{
    // No blank line required before this line as we have a line with an opening brace

    // For a new, unrelated topic use another blank line
    ReadInputFromUser();
}

// Blank line required before this line

Other topics

Namespaces

  • All public code should be in the global namespace. For this reason, it is important to choose global names that are unlikely to clash with other global names. It is, however, foreseeable that we may move place our code in the company namespace. In that case, all library code will be moved to the company namespace. Application code will not move.
  • Use anonymous namespaces in .cpp modules. When constructing types or functions that are specific to a translation unit, place those types and functions into an anonymous namespace.
  • Use 'package' namespaces for package-private code. It is often necessary to create types or functions that should not be known outside of a particular package, but are not local to a specific translation unit. In this case, a 'package' namespace may be used. The namespace name must be the same as the package name and then contain another namespace called detail. For example, the package-private namespace for the utils package would be utils::detail. If/when we implement the company namespace, this namespace will be part of the company namespace.

Bit masks

The recommended way of handling enumerations and bit masks is described below.

  • Create an enumeration to represent each of the values in the bit mask. Name the enumeration such that it is singular (not plural) and is representative of a single field in the enumeration (e.g. SearchField).
  • Create a typedef using the EnumBitField template class, defined in <utils/enumbitfield.h>, for your enumeration. Name the type using the plural of the enumeration (e.g. SearchFields).
  • Whenever the bit mask is required then use the EnumBitField typedef.
  • Whenever a single value is required then use the enumeration.

Null-terminated strings and buffers

Null terminated strings and buffers related to storing those are large sources of bugs due to off-by-one errors when storing null terminated strings into incorrectly allocated buffers. This section describes some conventions to try to avoid these problems, or at least make them identifiable during a code review.

  • Try not to use null terminated strings. In many cases, you should be able to use std::string instead. But if you're writing low-level code that has to deal at with null-terminated strings, then follow the conventions below.
  • Always pass the buffer size. When passing a buffer to a function than fills in a null-terminated string, pass an extra (size_t) parameter that represents the size of the buffer. The size of the buffer must include space for the null-terminator. Don't pass in the maximum length of the string. This makes it easy to write code like: GetString (buffer, sizeof (buffer)).
  • Use 'size' for buffer sizes and 'length' for string lengths. Variables that refer to the size of a buffer (including the null-terminator) must have the word 'size' in them e.g. the size variable in the function described above. Variables that refer to the length of a string (excluding the null-terminator) must have the word 'length' or 'len' in them. Using this convention makes it easy to spot violations.
