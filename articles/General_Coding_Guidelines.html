Introduction

This page contains general coding (and design) principles guidelines that should be adhered to across development languages and activities. Some of the examples may be focused on specific languages, but this should not detract from the overriding principles.


The "Do's"

This section discusses principles and techniques that must be followed and are considered to be best practices.

Follow good design principles

  • General design principles as discussed in this article.
  • Object oriented design principles as discussed in this article.

Use test driven development (TDD)

Our development process follows a test-driven or test-first methodology. This means that for every piece of code written, there should be unit test cases that test the code. These tests should be written before the actual code is written. The tests may be written (depending on the coding language), either in our perl test framework, or our C++ test framework.

You need to design your code in such a way that it is easy to write test cases. This generally means that you need to ruthlessly evaluate all your code's dependencies and remove those that are not required. Those that cannot be removed in this way, can often be removed by using Dependency Inversion[1].

Often, however, you will inherit code that did not follow the TDD practice. This code is dangerous because there is no safety net (i.e. tests) to tell you whether or not your change has broken an invariant. The article on unit test conventions provides more information regarding the standards and conventions for unit tests and how to deal with legacy code.

The "Don'ts"

This section discusses things that should not be done.

Copy and paste

Do not use copy-and-paste as a method of ruse. Let me repeat that: If you use copy-and-paste to reuse someone else's stuff in this department, you will buy drinks for everyone every Friday for a month. Copy-and-paste is the crudest form of reuse and results in multiple copies of the same thing. When it is code it also results in code bloat and makes it harder to fix bugs as these bugs need to be tracked down in every piece of copied code.

So what do you do if you find that someone else has something that you need and it mostly (or completely) does what you need it to do?

  1. Track down the author of the module/function/class/document/artifact.
  2. Between the 2 of you decide how the resource will be shared.
  3. Implement any changes to the resource to make it reusable to both of you.
  4. Modify, if necessary, any existing things that use the modified resource.
  5. Put the shared resource in a place that makes it logically usable by both parties.

Previous work "Must be right"

Just because something has been done in a particular way before, does not mean that it is a "standard" and must copied without questioning. If you need to do something and you are shown an example of how it was previously done, then question whether the previous design was correct (especially if you think it doesn't make sense or is inadequate). In many cases the person who did the work may have been under time pressure and did not have time to do it properly. Highlighting the issue may actually result in the person conceding that their original design was flawed and then fixing it, resulting in a better system for everyone. Remember, if it doesn't make sense to you, then chances are, it won't make sense to someone else.

If you are on the receiving end of the "why" question, make sure that you are convinced that you did it correctly when you did it. If you come to the conclusion that it was correct, then document the reasoning so that the question may be answered easily the next time round. If you conclude that it was wrong, then either fix the problem immediately, or log an improvement so that the issue is not forgotten.

Banned practices

The following practices are known to cause maintenance and other problems and must not be done without a very, very good reason:

  • Global variables. Global variables introduce unexpected dependencies and make writing test cases especially hard.
  • Singletons. Singletons are just glorified global variables and suffer the same problems as global variables. They should be avoided unless something specifically requires it, in which case it should be an implementation detail and the singleton should not be referenced in application code.
  • Clever code. Debugging is twice as hard as writing the code in the first place. Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it.
